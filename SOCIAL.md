## Day 10

### LinkedIn Post

Background jobs are the unsung hero of every serious web application.

Sending emails asynchronously. Processing image uploads. Generating reports. Syncing data with third-party APIs. Scheduling recurring tasks. All of these need to happen outside the HTTP request cycle — and doing them right is harder than it looks.

Grit solves this with asynq + Redis, and integrates it so seamlessly you might not even notice it's there until you need it.

```go
// Enqueue a job from anywhere in your Go code
err := jobs.Enqueue(ctx, &EmailJob{
    To:      "user@example.com",
    Subject: "Welcome to the app",
    Template: "welcome",
})
```

Your job handler:

```go
func (j *EmailJob) Process(ctx context.Context) error {
    return email.Send(ctx, j.To, j.Subject, j.Template)
}
```

That's it. The job is serialized, stored in Redis, and processed by a worker pool running alongside your API server. Retries, dead-letter queues, job scheduling — all handled by asynq.

You can monitor your queues directly in the Grit admin panel. No separate monitoring setup.

Production-grade background jobs in minutes, not days.

"Why not just use Next.js for everything?"

Next.js 14 with API routes is genuinely powerful. You can build a full-stack app with it. I have. Many developers have.

Here's where it starts to hurt:

**Concurrency.** Node.js is single-threaded. When you have CPU-intensive tasks — image processing, PDF generation, data transformations — you're fighting the event loop.

**Background jobs.** You end up adding BullMQ or similar, which means more infrastructure, more complexity, more things to monitor.

**Type safety across the stack.** You can do it in TypeScript, but it requires discipline and the right tooling to keep in sync.

**Performance at scale.** Node can handle a lot, but Go handles more with less memory. This matters when your bill arrives.

Grit's approach: use the right tool for each layer. Go handles the heavy lifting on the backend — HTTP routing (Gin), database (GORM), background jobs (asynq), caching (Redis). Next.js 14 handles what it's actually great at: building incredible user interfaces with React Server Components.

You get Go's concurrency for your API and Next.js's component model for your UI.

Best of both worlds. One framework.

### LinkedIn Post

Go vs the world — let's talk performance numbers.

This isn't a "my framework is better" post. It's a "here's what you're getting when you choose Go" post.

Some real-world benchmarks from the Go ecosystem:

- **Gin** (the HTTP router Grit uses) handles ~60,000 requests/second on modest hardware. Express.js handles roughly 15,000 on the same hardware.
- **Memory:** A Go HTTP server typically uses 10-30MB at idle. A Node.js server doing the same work often uses 100-200MB.
- **Goroutines:** Go can handle hundreds of thousands of concurrent goroutines. Threads in other runtimes are far more expensive.
- **Cold start:** Go binaries start in milliseconds. JVM-based apps often take seconds.

For small projects, these differences don't matter. You can scale Node or PHP to impressive heights with the right infrastructure.

But when your app grows, Go compounds in your favor. Lower memory = more instances per machine = lower hosting costs. Better concurrency = fewer servers needed.

Grit gives you all of this — packaged with the productivity of a full-stack framework.

High performance doesn't have to be hard to build with.

Redis caching in Grit — make your app fast by default.

Caching is one of those things that every developer knows they should be doing and few actually implement well. It's not hard, but it requires discipline.

Grit makes it embarrassingly easy.

```go
// Cache the result of an expensive query
posts, err := cache.Remember(ctx, "posts:featured", 15*time.Minute, func() ([]Post, error) {
    return h.postService.GetFeatured(ctx)
})

// Invalidate cache when data changes
cache.Forget(ctx, "posts:featured")

// Cache arbitrary values
cache.Set(ctx, "user:"+userID+":profile", profile, 1*time.Hour)
profile, err := cache.Get[Profile](ctx, "user:"+userID+":profile")
```

The cache service is backed by Redis (the same Redis instance used for background jobs). In development, it falls back gracefully if Redis isn't running.

On Grit Cloud, Redis is provisioned automatically alongside your application. You don't configure a connection string — it's wired in at deploy time.

Your application should be fast on day one. Grit makes that the default, not an afterthought.

## Day 25

### LinkedIn Post

Zero-downtime deployments in Grit Cloud — because downtime is unacceptable.

When you run `grit deploy`, here's what happens behind the scenes:

1. Your new container is built and pushed
2. The new version starts and health checks are run (`GET /health`)
3. Once the new instance passes health checks, traffic is shifted to it
4. The old instance is drained gracefully (in-flight requests complete)
5. The old container is terminated

Your users never see downtime. Background jobs that are in-flight continue to completion before the old worker exits. Asynq handles this gracefully — workers finish their current job and then shut down cleanly.

Database migrations run as part of the deployment pipeline with `grit db migrate`, before the new version starts serving traffic.

This is production-grade deployment behavior by default. You don't configure it. You don't think about it.

`grit deploy` and sleep well.
